# Tunix Distributed-RL Core Design

This document consolidates the architectural design and API contracts for Tunix's Distributed-RL framework. It is intended to serve as a comprehensive reference for LLM coding agents tasked with implementing the system.

## 1. Overview & Motivation

Tunix is evolving from a monolithic job architecture into a **Role-Based Isolation** architecture to unlock RL training at a **10K+ chip scale**. The framework decouples trajectory generation (rollout), trajectory scoring (inference), and gradient updates (trainer) into independently scalable micro-services. 

The system relies on a central **Orchestrator** acting as the control plane to manage execution flow, elasticity, weight synchronization, and fault recovery.

## 2. Core Components & Roles

The system is composed of several specialized roles, each with strict input/output boundaries to enable modularity and independent scaling:

* **Rollout Worker**: Generates sample trajectories via multi-turn interactions. Uses a pluggable sampler (e.g., vLLM or SGLang).
  * *Input*: `RequestInput` (prompt)
  * *Output*: `Trajectory`
* **Environment (Pool)**: A simulated sandbox enabling external tool interaction. Managed as a pool of instances.
  * *Input*: Action
  * *Output*: Observation & Reward
* **Agent**: The decision-maker inside the rollout loop that parses responses and generates actions.
* **Trajectory Collector Engine**: The component managing the overall control flow and lifecycle of a single trajectory generation across the Agent, Sampler, and Environment.
* **Inference / Reward Worker**: Processes trajectories generated by rollouts and scores them (e.g., computing advantages, reference log probs, or reward models).
  * *Input*: Raw trajectory micro-batch
  * *Output*: Scored trajectory micro-batch
* **Trainer Worker**: Consumes the scored training data to execute forward-backward passes and update model weights.
  * *Input*: `TrainData` micro-batch
  * *Output*: Metrics and updated weights
* **Trajectory Queue Manager**: Groups out-of-order trajectories, filters them (e.g., degenerate or stale data), and packs them into micro-batches for the Trainer.
* **Trajectory Store**: Asynchronous, persistent storage for in-flight and completed trajectories (following ATIF format) to enable stateless recovery.
* **Orchestrator**: The central Single Point of Failure (SPOF) control plane. Handles global step coordination, elasticity, fault tolerance, and weight sync triggers.
* **Rollout Manager**: A local control plane on the rollout worker handling the dispatch and concurrency of multiple trajectory requests.

## 3. Overall Orchestrator Worker Flow

The Orchestrator oversees the End-to-End training loop, which is divided into three dependent but partially parallelizable phases:

### Phase 1: Produce (Generating Trajectories)
- **1.A Data Loading**: The Orchestrator loads prompt data and the load balancer dispatches `num_generations` trajectory requests to the Rollout Workers.
- **1.B Rollout Execution**: Rollout workers receive prompts and execute the multi-turn agentic loop asynchronously. The `TrajectoryCollectEngine` orchestrates the Sampler and Environment steps. Completed trajectories are pushed to the **Trajectory Store** and the Orchestrator's queue.

### Phase 2: Consume (Consuming Trajectories & Training)
- **2.A Train Data Processing**: The **Trajectory Queue Manager** consumes the raw trajectories, groups them by prompt, filters out staleness/degenerate outputs, and packs them into raw micro-batches.
- **2.B Scoring**: The micro-batches are distributed to **Inference/Reward Workers** to run the reference model and compute advantages/KL-divergence.
- **2.C Training**: The scored micro-batches are sharded and distributed to the **Trainer Workers**. The Trainer accumulates gradients, applies updates, and (optionally) captures checkpoints. Upon completing the step, the Trainer signals the Orchestrator.

### Phase 3: Weight Sync
- **Trigger**: Once the Trainer signals step completion, the Orchestrator updates the global step and triggers a weight sync.
- **Barrier (Optional)**: The Orchestrator can wait for in-flight trajectories to finish, or utilize pause/resume semantics to avoid dropping work.
- **Transfer**: The Orchestrator provides transfer metadata. The weights are then synchronized from the Trainer to the Rollout workers (either via P2P using **Raiden**, or via a shared FileSystem/Lustre).
- **Resume**: Rollout workers signal completion and resume generating on the new policy.

## 4. Key Sub-Systems

### 4.1. Micro-Service Abstraction
Workers are runtime-agnostic (supporting Pathways and MCJax). Each micro-service acts as a "head node" managed by an abstract `Worker` lifecycle (`_infer_shapes`, `_initialize`, `_compile`, `_warmup`, `_start_loop`).

### 4.2. Generic Remote Execution (RPC/Actors)
Communication is handled via a generic RPC abstraction to separate ML logic from networking:
- **`RemoteExecutionServer`**: A daemon running on worker nodes (e.g., TPU pods) handling generic gRPC execution requests.
- **`ActorHandle`**: A reference to a specific stateful remote worker (e.g., Trainer) where execution must occur on the exact node holding optimizer state.
- **`ActorPool`**: A collection of stateless `ActorHandles` (e.g., Rollout workers) supporting load-balanced routing and broadcast methods.

### 4.3. Weight Sync (Raiden vs FileSystem)
Synchronizing 100GB+ of weights frequently is a bottleneck. The architecture supports multiple `WeightSyncController` implementations:
- **FileSystem (Checkpoint-based)**: Orchestrator dictates a shared CNS path. High latency overhead.
- **Raiden (Stream-based P2P over DCN)**: Trainer stages weights from TPU HBM to Host CPU. Rollout workers execute P2P transfers via zero-copy sockets, providing sub-linear fanouts and O(seconds) latency.

### 4.4. Fault Tolerance & Failure Recovery

The framework guarantees component-level fault tolerance to ensure localized failures do not crash the global training loop. The overarching principles are detecting failures via active error reporting and consistent heartbeats, and utilizing a **Background Restore Mechanism** to replace and re-sync workers asynchronously.

#### Failure Detection & Background Restore
- **Detection**: Handled via component-to-orchestrator error reporting (for retriable vs. non-retriable classification) and consistent heartbeats (to detect hardware failures).
- **Background Restore**: When a worker fails, it is de-registered by the Orchestrator. The underlying scheduler (e.g., GKE) provisions a replacement resource in the background. Once initialized and warmed up (with optimizations like compilation caching), the worker re-registers with the Orchestrator, and the load balancer resumes routing to it.

#### Component-Specific Recovery
- **Environment State Recovery**: Environment states can be massive. Instead of capturing full state checkpoints, the system persists the sequence of executed actions in the **Trajectory Store** and uses them to **deterministically replay and rebuild** the environment state from the beginning.
- **Rollout Worker Failure**:
  - For mid-generation trajectories: Rollout restores to the latest persisted state (either sampler or environment). If the latest sampler turn isn't persisted, the environment must be rewound and replayed to align with the oldest persisted state.
  - Load is reclaimed and reassigned to other healthy rollout workers, potentially pausing newer tasks to prioritize near-complete trajectories.
- **Inference & Trainer Worker Failure**:
  - The Orchestrator tracks the last micro-batch sent for inference/training.
  - If the Trainer fails, it recovers from the nearest full or sub-step checkpoint. The Orchestrator initiates a new weight sync and flushes stale metrics.
- **Orchestrator Failure (SPOF)**:
  - In v1, the Orchestrator is a Single Point of Failure. If it fails, the entire job restarts.
  - Recovery sequence: Rollback to the previous finished global batch based on the batch offset stored in the checkpoint metadata. Reload all generated trajectories from the Trajectory Store (v0 uses in-memory/on-disk Orchestrator storage for fast R/W), re-assign partials to rollout workers, and resume. High Availability (HA) via hot-standby is planned for v2.

### 4.5. Elasticity

Elasticity is a strict prerequisite at the 10K+ scale to mitigate resource bottlenecks and avoid resource underutilization. Scaling is handled by the **Elasticity Manager** (part of the Orchestrator) acting dynamically on various bottleneck scenarios.

#### Triggers and Mechanism
- **Triggers**: Can be **Manual** (user/multi-tenant controlled) or **Automated** (based on metrics like queue build-ups and TPU duty cycle).
- **Minimum Allocation Units (MAU)**: Roles are scaled horizontally in fixed chunks defined by their tensor constraints (`MAU = Tensor Parallelism * FSDP`). For example, Rollout is scaled by adding/removing fixed identical clusters of 8 chips (if TP=8, FSDP=1).
- **Background Reallocation**: To optimize downtime, new resources are acquired and initialized in the background. Once initialized, the Orchestrator executes a dynamic rendezvous to update routing state.
- **Timing (Per-role optimal)**: Instead of locking elasticity exclusively to global weight sync boundaries (which leads to idle time), the system performs dynamic mid-flight elasticity:
  - *Rollout removal* uses a timeout-driven approach: waiting for the current generation turn to complete before snapshotting and draining, avoiding long tail disruptions.

#### Elasticity Scenario Handling
The Elasticity Manager reacts to different pipeline bottlenecks autonomously:
- **Rollout Compute Bottleneck**: If TPUs cannot process initial prompts fast enough to keep envs busy (`rollout_tpu_duty_cycle > 85%`, `env_warm_pool_ready_count` high), **Scale UP Rollout**.
- **Simulation Capacity Bottleneck**: If TPUs are stalled waiting for Env observations (`rollout_tpu_duty_cycle < 20%`), **Scale UP Environment pool**.
- **Rollout Over-provisioning**: If generations outpace simulations entirely (`generation_inbound_queue_depth = 0`), **Scale DOWN Rollout**.
- **Scoring Bottleneck**: If the `trajectory_queue_depth` is rising but the Trainer is starved, **Scale UP Inference/Reward models**.
- **Trainer Saturation**: If the Trainer `tpu_duty_cycle > 95%` and queues are maxed, the system throttles the pipeline and **Scales DOWN Rollout/Env/Inference**. (Note: Trainer size is static in v1 due to the massive recompilation/resharding penalty of changing its topology mid-flight).

## 5. API Contracts & Core Classes

LLM Agents developing features for Tunix must adhere to the abstract base classes defined in `api_abcs.py`.

### Control Plane
- `Orchestrator`: The SPOF that runs `start_job()` to drive gradient accumulation, updates, weight sync, and checkpointing. 
- `WeightSyncController`: Manages the weight broadcast protocol.

### Workers
- `Worker`: Base interface defining the `start()` sequence and async background compilation.
- `RolloutWorker(Worker)`: Handles asynchronous generation (`async def generate`) and stream polling (`as_completed_stream`). Must handle `pause()`, `resume()`, and `update_weights()` events safely during async generation loops.
- `TrainerWorker(Worker)`: Wraps an `AbstractTrainer`. Executes `fwd_bwd`, `update`, and checkpoint operations. 
- `InferenceWorker(Worker)`: Exposes `score()` to augment raw trajectories with advantages.

### ML Core Separation
- `AbstractTrainer`: Pure JAX/Flax algorithmic core. **Knows nothing about networking, RPCs, or orchestration.** Operates on `TrainerPayload`. Maintains unreduced `MetricsBuffer` on-device to avoid host-device synchronization overhead.

### Payloads & Data Structures
- `RequestInput`: Config and prompt payload sent to rollout workers via a webhook/callback descriptor to prevent long-running gRPC timeouts.
- `Trajectory`: Base class mapping `RolloutOutput` to `TrainExample`, serialized via ATIF.
- `WeightSyncMetadata`: Defines transfer modes (file, raiden) and source endpoints.

## 6. Development Guidelines

0. **Modularity at the Core**: When implementing an algorithm or a component, make sure it is modular and can be used in different contexts.
1. **Separation of Concerns**: When implementing mathematical or algorithmic changes (e.g., GRPO loss), exclusively modify `AbstractTrainer` subclasses. When implementing distributed logic (e.g., peer-to-peer sync), modify the `Worker` and `Controller` layers.
2. **Actor Pattern Testing**: Use the `@remote(transport="inprocess")` abstraction when creating tests. This ensures your distributed logic is completely testable in a single Python process without network mocking.
3. **Async Rollouts**: Honor the `RolloutSyncLock` paradigm. Rollouts must be fully pause-able so that weight syncs do not corrupt trajectories in-flight.
4. **Metrics Processing**: Do not collect reduced arrays to the host during micro-batches. Accumulate via `WeightedMetric` inside `MetricsBuffer` on-device and reduce only at logging intervals.
